## LLM Integration Log

This document describes the current LLM-backed interrogation prototype for **Project-Mystery**.

---

## High-Level Architecture

- **Backend framework**: Python + Flask (`backend/app.py`)
- **LLM engine**: `MysteryLLM` (`backend/llm_engine.py`)
  - Base model: `HuggingFaceTB/SmolLM-135M-Instruct` (small, fast placeholder).
  - Optional character-specific LoRA adapters (via `peft.PeftModel`) for:
    - `intern_leo`
    - `dr_tara`
  - If adapters are not found on disk, the engine falls back to **persona system prompts** so the prototype still runs.
- **Vector database**: ChromaDB (`backend/database_manager.py`)
  - Persistent store under `database/chroma_data/`.
  - Collection: `case_kottayam_star`.
  - Seed facts:
    - The stolen **Kottayam Star** chip.
    - Leo’s temporary server-room access and access logs.
    - Dr. Tara’s revoked funding and potential motives.
- **SSE API**: `/interrogate` (POST)
  - Accepts an interrogation question, resolves relevant context from Chroma, routes to `MysteryLLM`, and streams tokens back via **Server-Sent Events**.

Data flow for one interrogation turn:

1. Frontend calls `POST /interrogate` with `character_id` and `message`.
2. Backend ensures ChromaDB is initialized and seeded (`initialize_case_data()`).
3. Backend queries Chroma for context with `retrieve_context(message, character_id)`.
4. Backend calls `MysteryLLM.generate_stream(...)` with the question, character ID, and retrieved facts.
5. Backend wraps the generated tokens into an SSE stream (`text/event-stream`) back to the browser.

## `/interrogate` Endpoint Specification

- **Method**: `POST`
- **Path**: `/interrogate`
- **Content-Type (request)**: `application/json`
- **Response type**: `text/event-stream` (SSE)

### Request Body

```json
{
  "player_id": "optional_string",
  "character_id": "string",
  "message": "string"
}
```

- **`player_id`** (optional):
  - Reserved for future use (session tracking, personalization).
- **`character_id`** (required):
  - The character being interrogated.
  - Currently supported values:
    - `"intern_leo"`
    - `"dr_tara"`
- **`message`** (required):
  - The investigator’s question / interrogation line in natural language.

### Validation & Error Responses

- If `character_id` or `message` is missing or empty:

  - **Status**: `400 Bad Request`
  - **Body**:

  ```json
  {
    "error": "character_id and message are required"
  }
  ```

### Back-end Behavior (Internal)

1. **Initialize case data**:
   - Calls `initialize_case_data()` (idempotent) to ensure that:
     - ChromaDB client is created at `database/chroma_data/`.
     - The `case_kottayam_star` collection exists and is seeded with the initial facts.
2. **Retrieve context**:
   - Calls:

   ```python
   context = retrieve_context(
       query=message,
       character_id=character_id,
   )
   ```

   - This runs a semantic query over the vector store and returns the most relevant facts for that character plus general case facts.
3. **LLM persona selection + generation**:
   - `MysteryLLM` attempts to load LoRA adapters (paths: `adapters/intern_leo`, `adapters/dr_tara`).
   - If an adapter is present and loads successfully, it calls `set_adapter(character_id)` before generation.
   - If not, it falls back to a persona-specific system prompt for that character.
   - It composes the final prompt using:
     - Character persona instruction.
     - Retrieved case context bullet list.
     - The investigator’s question.
   - Generates a streamed response with `TextIteratorStreamer`, chunked as text tokens.

## SSE Streaming Contract (Frontend Integration)

### Response Headers

- `Content-Type: text/event-stream`
- Typical SSE-related behaviors:
  - Connection is **long-lived**.
  - Data is sent as a sequence of SSE events until completion.

### Event Format

The backend yields events in the exact SSE text format:

- **Token events**:

```text
data: {token}\n
\n
```

Where `{token}` is the next piece of text generated by the LLM (may be a word, subword, or character).

- **Completion event**:

```text
data: [DONE]\n
\n
```

The `[DONE]` sentinel indicates that no more tokens will be sent for this request.

### Example SSE Stream (conceptual)

```text
data: I\n
\n
data:  don't\n
\n
data:  remember\n
\n
data:  leaving\n
\n
data:  the\n
\n
data:  door\n
\n
data:  open.\n
\n
data: [DONE]\n
\n
```

The frontend must:

- Concatenate `token` chunks in order to reconstruct the full reply.
- Stop reading when it receives the `data: [DONE]` event.

## Frontend Usage Guide (SSE)

Below are simple patterns for consuming the SSE stream. Exact implementation may vary depending on framework.

### Using `fetch` + ReadableStream (recommended)

```javascript
async function interrogate({ playerId, characterId, message }) {
  const response = await fetch("http://127.0.0.1:5000/interrogate", {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
    },
    body: JSON.stringify({
      player_id: playerId ?? null,
      character_id: characterId,
      message,
    }),
  });

  if (!response.ok || !response.body) {
    throw new Error("Interrogation request failed");
  }

  const reader = response.body.getReader();
  const decoder = new TextDecoder("utf-8");

  let fullText = "";

  while (true) {
    const { value, done } = await reader.read();
    if (done) break;

    const chunk = decoder.decode(value, { stream: true });

    // SSE frames might be split across chunks; buffer and parse by line.
    const lines = chunk.split("\n");
    for (const line of lines) {
      if (!line.startsWith("data:")) continue;
      const data = line.slice("data:".length).trim();
      if (!data) continue;

      if (data === "[DONE]") {
        // Stream finished
        return fullText;
      }

      // Append token to the running transcript
      fullText += data;
      // Optionally, update UI incrementally here.
    }
  }

  return fullText;
}
```

### Using EventSource (not ideal here)

The standard `EventSource` API assumes `GET` requests, whereas our API is a **POST** with a JSON body. You can only use `EventSource` if you:

- Change the backend to accept `GET` query parameters (not currently implemented), or
- Use a server-side proxy that converts a `GET` with query params into the `POST` against `/interrogate`.

For now, **prefer the `fetch` + `ReadableStream` pattern** shown above.

## Where to Extend Next

- **Additional characters**:
  - Add new `character_id` values and persona prompts in `MysteryLLM._build_persona_prompt()`.
  - Train and mount new LoRA adapters with distinct adapter names in `MysteryLLM.load_adapters()`.
- **Richer case data**:
  - Extend `initialize_case_data()` to include more detailed timelines, alibis, and contradictions.
  - Add new collections per case (e.g. `case_kottayam_star`, `case_other_city`).
- **Session-aware logic**:
  - Use `player_id` to track interrogation history and adjust responses based on previous questions.
